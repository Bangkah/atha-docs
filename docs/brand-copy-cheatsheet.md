# ATHA Brand Copy Cheatsheet

Use this guide to keep ATHA messaging consistent across landing page, docs, release notes, social posts, and campaign assets.

## Brand Positioning

ATHA is a workflow layer for pacman on Arch Linux.

ATHA helps users:

- inspect package decisions before execution
- simulate changes safely with dry-run
- review actions afterward with operation history

## Core One-Line Pitch

ATHA makes Arch Linux package operations safer, clearer, and reviewable.

## Extended Product Promise

Plan first, execute safely, and review everything after.

## Brand Voice

- Direct: use clear, concrete statements.
- Practical: focus on real workflows and real commands.
- Trustworthy: use claims that can be verified in product behavior.

## Primary Messaging Pillars

- Safety: plan and dry-run before execution.
- Transparency: show decision context before system changes.
- Auditability: keep operation history for post-action review.

## Pillar-to-Feature Mapping

- Safety -> plan mode, dry-run mode, explicit confirmations.
- Transparency -> decision context before package action.
- Auditability -> history timeline and summary review.

## Copy Formulas

Use these formulas to generate consistent product copy.

- Headline: Outcome + context
	Example: Safer package workflows for Arch Linux users.
- Subheadline: How it works in three verbs
	Example: Inspect decisions, simulate execution, review results.
- Feature line: Capability + user benefit
	Example: Plan mode shows package decision context before execution.

## Approved Headline Patterns

- ATHA makes Arch package operations safer by design.
- Plan first, execute safely, review everything.
- Safer package workflows for Arch Linux users.

## Approved Subheadline Patterns

- Inspect decisions before execution, simulate changes safely, and track what happened after.
- Workflow layer for pacman with plan mode, dry-run safety, and operation history.
- Built for Arch users who want explicit package actions and reliable audit trails.

- revolutionary
- perfect
- zero risk
- fully automatic decisioning

## Do and Don't

### Do

- Emphasize workflow outcomes: plan, dry-run, history.
- Mention Arch Linux and pacman context explicitly.
- Keep sentences short and action-oriented.
- Tie every claim to a product behavior users can verify.

### Don't

- Claim ATHA replaces pacman.
- Use vague superlatives without proof.
- Over-promise safety as absolute certainty.
- Mix brand casing (Atha, atha) in product copy.

## Naming and Casing Rules

- Product name: ATHA
- OS reference: Arch Linux
- Package manager reference: pacman
- Command examples: lowercase command form (for example: atha install --plan vim)

## Message Priority by Surface

- Hero: core promise + primary conversion CTA.
- Comparison sections: proof of capability differences.
- Trust signals: credibility and source transparency.
- FAQ and troubleshooting: reduce adoption friction.
- Footer and docs links: official references and continuity.

## Quick QA Checklist

- Is ATHA casing consistent everywhere?
- Is each promise tied to a real feature?
- Is the CTA mapped to awareness, consideration, or conversion?
- Is the sentence easy to verify from product behavior?
- Does copy mention Arch Linux and pacman when context needs it?
- Is tone direct, practical, and trustworthy?
